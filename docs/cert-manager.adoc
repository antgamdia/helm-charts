= Adding cert-manager for HTTPS

This guide will help you integrate cert-manager to enable HTTPS for your
Trento deployment. Two issuance strategies are covered:

* **Let's Encrypt** (recommended for production): automatically provisions
  publicly trusted certificates via ACME.
* **Self-signed** (suitable for testing or air-gapped environments): issues
  certificates signed by a cluster-local CA.

== Prerequisites

* Kubernetes cluster
* Helm installed
* Trento Helm chart

== Steps to Enable HTTPS (Let's Encrypt)

=== 1. Install cert-manager CRDs

First, install the cert-manager Custom Resource Definitions (CRDs):
[source,bash]
....
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.14.5/cert-manager.crds.yaml
....

=== 2. Add jetstack repo

Add the Jetstack Helm repository and update your local Helm chart
repository cache:
[source,bash]
....
helm repo add jetstack https://charts.jetstack.io
helm repo update
....

=== 3. Install cert-manager

Install cert-manager in the cert-manager namespace:
[source,bash]
....
helm install cert-manager jetstack/cert-manager --namespace cert-manager --create-namespace --version v1.14.5
....

=== 4. Generate a valid configuration for cert-manager

Use `+envsubst+` to substitute the templates with actual valid data
(make sure to replace the suggestions here with valid ones for your
deployment):
[source,bash]
....
export TRENTO_WEB_ORIGIN=example.com TRENTO_ADMIN_EMAIL=admin@example.com TRENTO_NAMESPACE=default TRENTO_INGRESS_CLASS=traefik
envsubst < certificate.tpl.yaml > certificate.yaml
envsubst < cluster-issuer.tpl.yaml > cluster-issuer.yaml
envsubst < override-values.tpl.yaml > override-values.yaml
....

NOTE: `+override-values.tpl.yaml+` configures TLS ingress for all three
Trento services: `+trento-web+` (path `/`), `+trento-wanda+` (path
`+/wanda+`), and `+trento-mcp-server+` (path `+/mcp+`). All three will
share the same `+trento-tls+` secret and the `+letsencrypt-production+`
cluster issuer.

=== 5. Apply cluster-issuer

Use `+envsubst+` to substitute the email for the domain certificate and
apply it afterwards:
[source,bash]
....
kubectl apply -f cluster-issuer.yaml
....

=== 6. Apply certificate

Edit `+certificate.yaml+` to update the namespace and FQDN, and apply it
afterwards:
[source,bash]
....
kubectl apply -f certificate.yaml
....

=== 7. Deploy Trento with cert-manager support

Navigate to the Trento Helm chart directory and deploy it with
cert-manager support enabled. Remember to change the admin password:
[source,bash]
....
cd helm-charts-rolling/charts/trento-server
helm dependency update
helm upgrade -i trento --wait . \
--set trento-web.adminUser.password="somepassword" \
-f override-values.yaml
....

== Alternative: Self-Signed Certificates (Testing / Air-Gapped)

For environments without public DNS or internet access (e.g. local CI,
internal staging), a self-signed `+ClusterIssuer+` is provided in
`+hack/cert-manager/selfsigned-issuer.yaml+`.

=== 1. Apply the self-signed issuer

[source,bash]
....
kubectl apply -f selfsigned-issuer.yaml
....

This creates a `+ClusterIssuer+` named `+selfsigned-issuer+`.

=== 2. Create a certificate referencing the self-signed issuer

Edit `+certificate.tpl.yaml+` (or create a custom `+Certificate+`
resource) and set `+issuerRef.name: selfsigned-issuer+` before
applying it:

[source,yaml]
....
spec:
  issuerRef:
    name: selfsigned-issuer
    kind: ClusterIssuer
....

[source,bash]
....
kubectl apply -f certificate.yaml
....

=== 3. Deploy Trento with override values

Use the same `+override-values.yaml+` generated from
`+override-values.tpl.yaml+`. The ingress annotations reference
`+letsencrypt-production+` by default; update the
`+cert-manager.io/cluster-issuer+` annotation in each ingress entry to
`+selfsigned-issuer+` when using self-signed certificates.

[source,bash]
....
helm upgrade -i trento --wait . \
--set trento-web.adminUser.password="somepassword" \
-f override-values.yaml
....
